AbstractQueuedSynchronizer

发表于 2020-07-23 | 分类于 详解java并发

概念

队列同步器AbstractQueuedSynchronizer(以下简称AQS)是构建锁和其他同步组件的基础结构，它使用了一个state变量来表示同步状态，还有一些系列方法来管理这个同步状态，内置FIFO队列来完成资源获取线程的排队工作。

重写方法

如果你有认真看过并发包下ReentrantLock、ReentrantReadWriteLock和CountDownLatch等一系列同步组件源码，会发现这些组件都是通过继承AbstractQueuedSynchronizer并实现它的抽象方法来管理同步状态，同步组件和AQS区别点在于：

  • 同步组件是面向程序员，也就是锁的使用者的，它提供了使用者和锁交互的接口，隐藏了实现细节；
  • AQS是面对同步组件的，它简化了锁的实现方式，屏蔽了同步状态管理，线程排队，等待和唤醒等底层操作。

AQS基于模板方式设计，也就是使用者继承AQS，重写指定方法，并将AQS组合在自定义的同步组件中，当调用AQS的模板方法时，最终会调用到自定义同步组件的方法。

同步队列CLH

在详解内部方法之前，还需要了解CLH这个同步队列的概念。

AQS依赖内部的同步队列（一个FIFO双向队列）来完成同步状态的管理，当前线程获取同步状态失败时，AQS会将当前线程以及等待状态等信息构造成为一个节点（Node）并将其加入同步队列，同时会阻塞当前线程，当同步状态释放时，会把同步队列的首节点中的线程唤醒，使其再次尝试获取同步状态。 同步队列中的节点（Node）用来保存获取同步状态失败的线程引用、等待状态以及前驱和后继节点

节点Node

也就是说我们调用前面那些独占式或者共享式的方法时，肯定会有成功失败的案例，对于失败的，就会把当前线程的信息构造成一个节点Node(比如，独占式获取，那么这个Node就是Node.EXCLUSIVE),加入到队列中。当这个线程在队列中了，肯定会想着怎么出去，或者我不等待了，那么就是根据waitStatus这个状态来判定线程行为。
